What is Claude Code?
An agentic coding environment. Unlike a chatbot, it reads your files, runs commands, makes changes, and autonomously works through problems while you watch, redirect, or step away entirely.

The #1 Constraint: Context Window

Almost every best practice boils down to one thing: the context window fills up fast, and performance degrades as it fills.

Every message, every file Claude reads, and every command output shares the same limited budget. A single debugging session might consume tens of thousands of tokens.

✅ Core Practice #1

Give Claude a Way to Verify Its Work

Include tests, screenshots, or expected outputs so Claude can check itself. This is the single highest-leverage thing you can do.

Claude performs dramatically better when it can verify its own work. Without clear success criteria, it might produce something that looks right but doesn't actually work. You become the only feedback loop, and every mistake requires your attention.

Verification Strategies

Strategy❌ Without Verification✅ With Verification
Provide test cases"implement a function that validates email addresses""write a validateEmail function. test cases: user@example.com → true, invalid → false. run the tests after implementing."
Verify UI visually"make the dashboard look better""[paste screenshot] implement this design. take a screenshot of the result. list differences and fix them."
Address root causes"the build is failing""the build fails with: [paste error]. fix it and verify the build succeeds. address root cause, don't suppress."
💡
Tip
UI changes can be verified using the Claude in Chrome extension. It opens new tabs in your browser, tests the UI, and iterates until the code works.

What Makes Good Verification?

✅ Good verification
  • Automated test suite Claude can run
  • Expected outputs Claude can diff against
  • A linter or type-checker with exit codes
  • Screenshots for UI changes
  • Bash commands that check conditions
❌ Poor verification
  • "Looks right to me" (you're the only loop)
  • No tests at all
  • Suppressed errors / warnings
  • Only checking happy paths
  • Assuming without running
⚠️
Important
Invest in making your verification rock-solid. The quality of your verification directly determines the quality of Claude's output.

Normal Mode

⚠️
When to Skip Planning
Plan Mode adds overhead. For small tasks (fixing a typo, adding a log line, renaming a variable) — just ask Claude directly. Planning pays off when you're uncertain about the approach, the change touches multiple files, or you're unfamiliar with the code.

Plan vs. Just Do It

Use Plan Mode when…Skip planning when…
Uncertain about the approachYou could describe the diff in one sentence
Change modifies multiple filesThe task is small and obvious
Unfamiliar with the codeYou've done this exact thing before
Feature could be built several waysIt's a typo / rename / log line
📌 Core Practice #3

Provide Specific Context

The more precise your instructions, the fewer corrections you'll need. Reference files, mention constraints, point to existing patterns.

Prompting Strategies

Strategy❌ Vague✅ Specific
Scope the task
File, scenario, test preferences		"add tests for foo.py""write a test for foo.py covering the edge case where the user is logged out. avoid mocks."
Point to sources
The authoritative source		"why does ExecutionFactory have a weird api?""look through ExecutionFactory's git history and summarize how its api came to be"
Reference patterns
Existing code to follow		"add a calendar widget""look at HotDogWidget.php to understand widget patterns, then build a calendar widget. no new libraries."
Describe symptom
Symptom + location + done criteria		"fix the login bug""users report login fails after session timeout. check src/auth/, especially token refresh. write a failing test, then fix."
💡
Vague prompts have a place
For exploration, vague is fine. "What would you improve in this file?" surfaces things you wouldn't have thought to ask. But for execution — be specific.

📄 Configuration

Writing an Effective CLAUDE.md

Read at the start of every conversation. Gives Claude persistent context it can't infer from code alone.

💡
Quick Start
Run /init to auto-generate a starter CLAUDE.md. It detects build systems, test frameworks, and code patterns — then refine over time.

Example CLAUDE.md

CLAUDE.md
# Code style
- Use ES modules (import/export) syntax, not CommonJS (require)
- Destructure imports when possible (eg. import { foo } from 'bar')

# Workflow
- Typecheck when you're done making a series of code changes
- Prefer running single tests, not the whole suite, for performance

# When compacting, always preserve:
- The full list of modified files
- Any test commands we've been running

Include vs. Exclude

✅ Include
  • Bash commands Claude can't guess
  • Code style rules that differ from defaults
  • Testing instructions and preferred test runners
  • Repository etiquette (branch naming, PR conventions)
  • Architectural decisions specific to your project
  • Developer environment quirks (required env vars)
  • Common gotchas or non-obvious behaviors
❌ Exclude
  • Anything Claude can infer from reading code
  • Standard language conventions Claude knows
  • Detailed API docs (link instead)
  • Information that changes frequently
  • Long explanations or tutorials
  • File-by-file codebase descriptions
  • Self-evident rules like "write clean code"

Diagnosing CLAUDE.md Problems

SymptomCauseFix
Claude ignores rulesFile too long — rules get lostPrune ruthlessly. Convert rules to hooks.
Claude asks questions it should knowPhrasing is ambiguousRewrite for clarity. Test by observing behavior.
Rules followed inconsistentlyPriority unclearAdd "IMPORTANT:" or "YOU MUST" emphasis.

File Locations

LocationScopeUse Case
~/.claude/CLAUDE.mdAll sessionsPersonal preferences across all projects
./CLAUDE.mdProjectTeam conventions — check into git
./parent/CLAUDE.mdParent dirsMonorepo shared rules
./child/CLAUDE.mdSubdirectoryArea-specific rules, loaded on demand
🚨
Critical Warning
Bloated CLAUDE.md files cause Claude to ignore your actual instructions. Keep it under 200 lines. For each line ask: "Would removing this cause Claude to make mistakes?" If not, cut it.
🔐 Configuration

Configure Permissions

Reduce interruptions while staying in control. Three approaches — choose based on how much you trust the task.

🤖

Auto Mode

A classifier reviews commands and blocks only risky ones: scope escalation, unknown infrastructure, hostile-content actions. Best when you trust the direction.

📋

Permission Allowlists

Pre-permit specific safe tools: npm run lint, git commit. Use /permissions to manage.

📦

Sandboxing

OS-level isolation that restricts filesystem and network access. Claude works freely within defined boundaries.

Permission Modes

ModeFile EditsCommandsBest For
defaultAskAskInteractive development
acceptEditsAuto-approveAskTrusted editing sessions
planBlockedAskResearch / analysis only
autoClassifierClassifierAutonomous operation
bypassPermissionsAuto-approveAuto-approveFully trusted environments

Hooks — Deterministic Automation

Unlike CLAUDE.md instructions (advisory), hooks guarantee the action happens every time, with zero exceptions.

💡
Tip
Use hooks for actions that must happen every time with zero exceptions. Claude can write hooks for you — just describe what you need.
"Write a hook that runs eslint after every file edit"
"Write a hook that blocks writes to the migrations folder."

Hook Event Points

EventWhen it firesCan block?Common use
PreToolUseBefore any tool executesYesBlock dangerous commands, validate inputs
PostToolUseAfter tool succeedsNoAuto-format, audit log, run linter
SessionStartSession begins / after compactionYesRe-inject critical context
UserPromptSubmitWhen you submit a messageYesInject dynamic context
StopClaude finishes respondingYesVerify completion criteria

CLAUDE.md vs. Hooks

CLAUDE.mdHooks
ExecutionAdvisory — Claude may not followDeterministic — always runs
Can block actionsNoYes (PreToolUse)
Best forConventions, style, project contextMandatory automation
Example"Use 2-space indentation""Run eslint after every edit"

Skills & Custom Subagents

Extend Claude with domain knowledge, reusable workflows, and isolated specialist agents.

Skills

💡
Tip
Create SKILL.md files in .claude/skills/. Claude applies them automatically when relevant, or you invoke them with /skill-name.

Domain Knowledge Skill

.claude/skills/api-conventions/SKILL.md
---
name: api-conventions
description: REST API design conventions for our services
---
# API Conventions
- Use kebab-case for URL paths
- Use camelCase for JSON properties
- Always include pagination for list endpoints
- Version APIs in the URL path (/v1/, /v2/)

Repeatable Workflow Skill

.claude/skills/fix-issue/SKILL.md
---
name: fix-issue
description: Fix a GitHub issue
disable-model-invocation: true
---
Analyze and fix GitHub issue: $ARGUMENTS.

1. Use `gh issue view` to get issue details
2. Search codebase for relevant files
3. Implement the necessary changes
4. Write and run tests to verify
5. Create a descriptive commit and open a PR

Run /fix-issue 1234 to invoke it. Use disable-model-invocation: true for workflows with side effects you want to trigger manually.

Custom Subagents

💡
Tip
Subagents run in their own context with their own allowed tools — keeping your main conversation clean for implementation.
.claude/agents/security-reviewer.md
---
name: security-reviewer
description: Reviews code for security vulnerabilities
tools: Read, Grep, Glob, Bash
model: opus
---
You are a senior security engineer. Review code for:
- Injection vulnerabilities (SQL, XSS, command injection)
- Authentication and authorization flaws
- Secrets or credentials in code
- Insecure data handling

Provide specific line references and suggested fixes.

Let Claude Interview You

For larger features, have Claude interview you first — it asks about things you might not have considered.

💡
Tip
Start with a minimal prompt and ask Claude to interview you using the AskUserQuestion tool. Claude asks about technical implementation, UI/UX, edge cases, and tradeoffs.

The Interview Prompt

I want to build [brief description]. Interview me in detail
using the AskUserQuestion tool.

Ask about technical implementation, UI/UX, edge cases, concerns,
and tradeoffs. Don't ask obvious questions, dig into the hard parts
I might not have considered.

Keep interviewing until we've covered everything, then write a
complete spec to SPEC.md.
⚠️
Start Fresh for Implementation
Once the spec is complete, start a fresh session to execute it. The new session has clean context focused entirely on implementation, and you have a written spec to reference.

🔄 Session Management

Course-Correct Early and Often

The best results come from tight feedback loops. Correct Claude as soon as you notice it going off track.

Key Commands

ActionCommandEffect
Stop mid-actionEscClaude stops immediately. Context preserved. You can redirect.
RewindEscEsc or /rewindOpens rewind menu: restore conversation, code, or both.
Undo changes"Undo that"Claude reverts its most recent changes.
Reset context/clearResets context window entirely. Start fresh.
🚨
The Two-Corrections Rule
If you've corrected Claude more than twice on the same issue, the context is polluted with failed approaches. Run /clear and start fresh with a better prompt. A clean session with a better prompt almost always wins.
🎯

Tight feedback

Correct immediately when you notice the wrong direction. Don't let Claude go far down a bad path.

🔁

Try risky things

Checkpoints persist across sessions. Tell Claude to try something speculative — if it fails, rewind and try differently.

💾 Session Management

Checkpoints & Session Resume

Every action Claude makes creates a checkpoint. Conversations persist locally. Use this to your advantage.

💡
Tip
Double-tap Esc or run /rewind to open the rewind menu. Checkpoints persist across sessions — close your terminal and still rewind later.

Rewind Options

💬

Conversation only

Go back to a previous conversation state without touching code.

📝

Code only

Revert files without losing conversation context.

🔄

Both

Full rollback to any previous state.

📋

Summarize from here

Condense context from a selected message forward.

🚨
Not a Replacement for Git
Checkpoints only track changes made by Claude, not external processes. Always commit real work to git.

Resume Conversations

bash
claude --continue    # Resume the most recent conversation
claude --resume      # Select from recent conversations
💡
Name Your Sessions
Use /rename to give sessions descriptive names like oauth-migration or debugging-memory-leak. Treat sessions like branches — different workstreams get separate, persistent contexts.
⚙ Automation

Non-Interactive Mode

Run Claude in CI pipelines, pre-commit hooks, and automated workflows with claude -p.

bash
# One-off query
claude -p "Explain what this project does"

# Structured output for scripts
claude -p "List all API endpoints" --output-format json

# Streaming for real-time processing
claude -p "Analyze this log file" --output-format stream-json

# Fully autonomous
claude --permission-mode auto -p "fix all lint errors"

# Pipe data in
cat error.log | claude -p "What caused these errors?"

Output Formats

FormatUse Case
text (default)Human-readable output
jsonParse results programmatically in scripts
stream-jsonReal-time processing of long-running outputs
⚠️
Debug vs. Production
Use --verbose for debugging during development. Turn it off in production to reduce noise.

Parallel Sessions

Multiply your output by running multiple Claude sessions simultaneously — for parallel work and quality-focused workflows.

🖥

Desktop App

Manage multiple local sessions visually. Each gets its own isolated git worktree.

☁

Web (cloud)

Run on Anthropic's cloud infrastructure in isolated VMs. No local setup needed.

👥

Agent Teams

Automated coordination of multiple sessions with shared tasks and a team lead.

Writer / Reviewer Pattern

A fresh context improves review quality — Session B won't be biased toward code it didn't write:

Session A (Writer)Session B (Reviewer)
"Implement a rate limiter for our API"
"Review @src/middleware/rateLimiter.ts. Look for edge cases, race conditions, and consistency with existing middleware."
"Here's the review: [Session B output]. Address these issues."

Writer / Tester Pattern

One Claude writes tests, another writes code to pass them — enforces TDD naturally:

Session A (Test Writer)Session B (Implementer)
"Write comprehensive tests for a rate limiter. Define the interface first."
"Implement a rate limiter that passes these tests: @tests/rateLimiter.test.ts"
📡 Automation

Fan-Out Patterns

Distribute work across many parallel Claude invocations for large-scale migrations, analyses, or transformations.

1

Generate a task list

Have Claude list all files that need changing.

bash
claude -p "list all Python files that import the old logger" \
  --output-format json > files.txt
2

Script the loop

bash
for file in $(cat files.txt); do
  claude -p "Migrate $file from old_logger to new_logger.
             Return OK or FAIL." \
    --allowedTools "Edit,Bash(git commit *)"
done
3

Test on a few, then scale

Refine your prompt based on the first 2-3 files. The --allowedTools flag restricts what Claude can do — essential for unattended runs.

Pipeline Integration

bash
claude -p "<your prompt>" --output-format json | your_command
⚠️
Auto Mode in Headless Runs
For -p flag, auto mode aborts if the classifier repeatedly blocks — there's no user to fall back to. Design prompts to avoid scope escalation.
🚫 Mastery

Avoid Common Failure Patterns

Recognize these time-wasting anti-patterns early. Each has a fast fix.

🗑 The Kitchen Sink Session

❌ You start with one task, ask Claude something unrelated, then go back to the first. Context full of irrelevant information.
✅ Fix: /clear between unrelated tasks.

🔁 Correcting Over and Over

❌ Claude does something wrong, you correct it, still wrong, you correct again. Context polluted with failed approaches.
✅ Fix: After two failed corrections, /clear and write a better initial prompt incorporating what you learned.

📜 The Over-Specified CLAUDE.md

❌ CLAUDE.md is too long. Claude ignores half of it because important rules get lost in noise.
✅ Fix: Ruthlessly prune. If Claude already does it correctly without the instruction, delete it or convert to a hook.

🤷 The Trust-Then-Verify Gap

❌ Claude produces plausible-looking code that doesn't handle edge cases. You shipped it without verifying.
✅ Fix: Always provide verification (tests, scripts, screenshots). If you can't verify it, don't ship it.

🌊 The Infinite Exploration

❌ You ask Claude to "investigate" without scoping it. Claude reads hundreds of files, filling the context.
✅ Fix: Scope investigations narrowly, or use subagents so exploration doesn't consume your main context.

🧠 Mastery

Develop Your Intuition

The patterns in this guide are starting points. Over time, you'll develop intuition that no guide can capture.

Sometimes you should let context accumulate — when you're deep in one complex problem and the history is valuable. Sometimes skip planning because the task is exploratory. Sometimes a vague prompt is exactly right because you want to see how Claude interprets the problem before constraining it.

ℹ️
The Meta-Pattern
Pay attention to what works. When Claude produces great output, notice what you did: the prompt structure, the context you provided, the mode you were in. When Claude struggles, ask why. Over time, you'll know when to be specific and when to be open-ended.

Build Your Mental Model

✨

When Claude succeeds

Notice what you did: prompt structure, context provided, mode. Add those patterns to CLAUDE.md.

🔍

When Claude struggles

Ask why. Context noisy? Prompt vague? Task too big for one pass?

🧪

Experiment freely

Try risky approaches. Checkpoints let you undo. Exploration is cheap when you can rewind.

CLAUDE.md compounds

Every lesson learned added to CLAUDE.md pays dividends in every future session.
